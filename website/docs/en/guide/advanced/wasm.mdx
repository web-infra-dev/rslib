# Wasm

Rslib natively supports WebAssembly (WASM) modules, allowing you to directly import and use `.wasm` files in your project using the ESM import mechanism defined by the [WebAssembly ESM Integration](https://github.com/WebAssembly/esm-integration) proposal.

:::warning

When importing `.wasm` modules using WebAssembly ESM Integration, only ESM output is supported, so [format](/config/lib/format) must be set to `'esm'` (the default).

:::

## Import WASM modules

Rslib supports the following import forms for `.wasm` modules.

### Static import and export

Import a `.wasm` module using standard ESM syntax and access its instantiated exports:

```ts
import { add } from './add.wasm';
import * as wasm from './add.wasm';
import './add.wasm';

export { add } from './add.wasm';
export * from './add.wasm';
export * as wasm from './add.wasm';
```

### Dynamic import

```ts
const wasm = await import('./add.wasm');
```

### Source phase import

You can use [Source Phase Imports](https://github.com/tc39/proposal-source-phase-imports) to obtain a compiled `WebAssembly.Module` and instantiate it manually with a custom import object:

```ts
import source addModule from './add.wasm';

const { instance } = await WebAssembly.instantiate(addModule, {
  env: { now: Date.now },
});
```

You can also use `import.source()` to obtain a compiled `WebAssembly.Module` dynamically:

```ts
const addModule = await import.source('./add.wasm');
```

:::note

TypeScript cannot currently parse `import source` or `import.source()`. Use these forms in JavaScript files, or choose a toolchain that supports them.

:::

## Output modes

You can use [lib.wasm.mode](/config/lib/wasm#wasmmode) to select the output mode for `.wasm` modules. The appropriate mode depends on the [bundle](/config/lib/bundle) configuration and whether the consumer supports WebAssembly ESM Integration:

- When `bundle` is `true`, only `compile` mode is available.
- When `bundle` is `false`:
  - If the output is consumed by a downstream build tool that supports WebAssembly ESM Integration (such as Rsbuild), or a target runtime with native support (such as [Node.js](https://nodejs.org/api/esm.html#wasm-modules) versions matching `^22.19.0 || >=24.5.0`), use `preserve` mode (the default). The consumer is responsible for resolving and loading the `.wasm` modules.
  - If the consumer does not support WebAssembly ESM Integration, or you do not want to rely on it to handle `.wasm` modules, use `compile` mode.

### Compile mode

Rslib parses each `.wasm` module, generates the JavaScript glue code required to load and instantiate it, and emits its binary as a static asset.

The binary is emitted under [output.distPath.wasm](/config/rsbuild/output#outputdistpath) (defaults to `static/wasm`) with a content-hashed filename:

<Tabs>
<Tab label="bundle">

```
dist
├── index.js
└── static
    └── wasm
        └── [contenthash].module.wasm
```

</Tab>
<Tab label="bundleless">

```
dist
├── index.js
├── utils.js
└── static
    └── wasm
        └── [contenthash].module.wasm
```

</Tab>
</Tabs>

The generated loading code depends on [output.target](/config/rsbuild/output#outputtarget):

- `web`: Loads `.wasm` files with `fetch`.
- `node`: Loads `.wasm` files with asynchronous Node.js file system APIs.

### Preserve mode

Rslib keeps imports of `.wasm` modules in the JavaScript output and emits their binaries unchanged.

The `.wasm` file is emitted using the same source-relative path and original filename:

```
dist
├── index.js
├── utils.js
└── add.wasm
```

```js title="dist/index.js"
import { add } from './add.wasm';

export { add };
```

:::note Path constraints when .wasm imports JavaScript

In `preserve` mode, `.wasm` filenames are not affected by [output.filenameHash](/config/rsbuild/output#outputfilenamehash).

Rslib updates `.wasm` imports in JavaScript to point to the emitted files, but does not modify imports inside the WebAssembly binary. If a `.wasm` module imports JavaScript (for example, when using wasm-bindgen's `bundler` target), avoid the following configurations, which can change the output path or filename of the JavaScript files:

- [output.distPath.js](/config/rsbuild/output#outputdistpath)
- [output.filename.js](/config/rsbuild/output#outputfilename)
- [autoExtension](/config/lib/auto-extension)

:::

## Use wasm-bindgen

[`wasm-bindgen`](https://wasm-bindgen.github.io/wasm-bindgen/) is a tool for building WebAssembly libraries with Rust. The `--target` option generates output for different runtime environments.

Rslib supports the following targets:

- [`bundler`](#bundler) (recommended)
- [`module`](#module)
- [`web` / `experimental-nodejs-module`](#web--experimental-nodejs-module)

### `bundler`

The JavaScript glue imports the `.wasm` file as an ES module, and the `.wasm` module imports back into the glue for its imports object. This two-way dependency means Preserve mode is subject to the [path constraints described above](#preserve-mode): the glue's relative path and filename must remain unchanged. Compile mode is not affected.

### `module`

The glue JavaScript file imports the `.wasm` file with the [source phase import](#source-phase-import) syntax and constructs the imports object entirely in JavaScript. Since the `.wasm` file does not import JavaScript, the output path and filename of the JavaScript file do not need to be preserved, and both Compile and Preserve modes work out of the box.

### `web` / `experimental-nodejs-module`

The glue JavaScript file loads the `.wasm` file through `new URL('./pkg.wasm', import.meta.url)`. Rslib treats it as a static asset instead of a WebAssembly ESM module.

`wasm.mode` does not apply in this case.

## Type declaration

TypeScript does not provide built-in module declarations for `.wasm` files. If you use TypeScript, add a declaration file next to the `.wasm` file using the `.d.wasm.ts` extension, and enable [`allowArbitraryExtensions`](https://www.typescriptlang.org/tsconfig/allowArbitraryExtensions.html) in `tsconfig.json`:

```ts title="src/add.d.wasm.ts"
export function add(a: number, b: number): number;
```

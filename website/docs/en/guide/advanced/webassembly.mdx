# WebAssembly

Rslib supports WebAssembly (`.wasm`) modules used with the [WebAssembly ESM Integration](https://github.com/WebAssembly/esm-integration) proposal syntax.

:::warning

Source code that uses the WebAssembly ESM Integration proposal syntax can only be built as ESM output, so [format](/config/lib/format) must be `'esm'` (the default value).

:::

## Import WASM modules

Rslib supports the following import forms for `.wasm` modules.

### Static import and export

Import a `.wasm` module using standard ESM syntax and access its instantiated exports:

```ts
import { add } from './add.wasm';
import * as wasm from './add.wasm';
import './add.wasm';

export { add } from './add.wasm';
export * from './add.wasm';
export * as wasm from './add.wasm';
```

### Dynamic import

```ts
const wasm = await import('./add.wasm');
```

### Source phase import

[`import source`](https://github.com/tc39/proposal-source-phase-imports) is part of WebAssembly ESM Integration. It imports a compiled `WebAssembly.Module`, allowing you to instantiate it manually with a custom import object:

```ts
import source addModule from './add.wasm';

const { instance } = await WebAssembly.instantiate(addModule, {
  env: { now: Date.now },
});
```

Dynamic source phase import is also supported through `import.source()`:

```ts
const addModule = await import.source('./add.wasm');
```

:::note

TypeScript does not currently parse `import source` or `import.source()`. Write source phase imports in JavaScript files, or use a toolchain that supports the syntax.

:::

## Output

Control how `.wasm` modules are emitted with [lib.wasm.mode](/config/lib/wasm#wasmmode):

- **Compile**: Rslib parses the `.wasm` module, generates JavaScript glue code that loads and instantiates WebAssembly, and emits the binary as a static asset. Consumers do not need to support ESM Integration.
- **Preserve**: Rslib keeps the `.wasm` import in the output JavaScript and emits the binary as-is. The downstream build tool or runtime is responsible for resolving and loading the `.wasm` module. Preserve mode requires [bundle](/config/lib/bundle) to be `false`.

| bundle  | Default mode | Available modes          |
| ------- | ------------ | ------------------------ |
| `true`  | `compile`    | `compile` only           |
| `false` | `preserve`   | `compile` or `preserve`  |

If the output is consumed directly by a runtime such as a browser, use Compile mode. If a bundleless output will be processed by a build tool that supports ESM Integration (such as Rsbuild), or if the target runtime (such as [Node.js](https://nodejs.org/api/esm.html#wasm-modules) `^22.19.0 || >=24.5.0`) supports this feature natively, you can use Preserve mode.

### Compile mode

The binary is emitted under [output.distPath.wasm](/config/rsbuild/output#outputdistpath) (defaults to `static/wasm`) with a content-hashed filename:

<Tabs>
<Tab label="bundle">

```
dist
├── index.js
└── static
    └── wasm
        └── [contenthash].module.wasm
```

</Tab>
<Tab label="bundleless">

```
dist
├── index.js
├── utils.js
└── static
    └── wasm
        └── [contenthash].module.wasm
```

</Tab>
</Tabs>

The generated loading runtime is derived from [output.target](/config/rsbuild/output#outputtarget):

- For `web` targets, `.wasm` is loaded with `fetch`.
- For `node` targets, `.wasm` is loaded from the file system with async Node.js APIs.

### Preserve mode

The `.wasm` file is emitted using the same source-relative path and original filename:

```
dist
├── index.js
├── utils.js
└── add.wasm
```

```js title="dist/index.js"
import { add } from './add.wasm';

export { add };
```

The filename is not affected by [output.filenameHash](/config/rsbuild/output#outputfilenamehash).

Rslib updates JavaScript imports to follow the emitted `.wasm` files. However, imports inside the WebAssembly binary are kept as-is — if a `.wasm` file imports JavaScript (e.g. wasm-bindgen `bundler` target), avoid configurations that change the relative path or filename of the JavaScript output:

- [output.distPath.js](/config/rsbuild/output#outputdistpath)
- [output.filename.js](/config/rsbuild/output#outputfilename)
- [autoExtension](/config/lib/auto-extension)

## Use wasm-bindgen

[`wasm-bindgen`](https://wasm-bindgen.github.io/wasm-bindgen/) is a tool for building WebAssembly libraries with Rust. The `--target` option generates output for different runtime environments.

Rslib supports the following targets:

- [`bundler`](#bundler) (recommended)
- [`module`](#module)
- [`web` / `experimental-nodejs-module`](#web--experimental-nodejs-module)

### `bundler`

The JavaScript glue imports the `.wasm` file as an ES module, and the `.wasm` module imports back into the glue for its imports object. This two-way dependency means Preserve mode is subject to the [reverse-import constraints](#preserve-mode) — the glue's relative path and filename must remain unchanged. Compile mode is not affected.

### `module`

The glue JavaScript file imports the `.wasm` file with the [source phase import](#source-phase-import) syntax and constructs the imports object entirely in JavaScript. Since the `.wasm` file does not import JavaScript back, there are no reverse-import constraints — no filename or path configuration to worry about, and both Compile and Preserve modes work out of the box.

### `web` / `experimental-nodejs-module`

The glue JavaScript file loads the `.wasm` file through `new URL('./pkg.wasm', import.meta.url)`. Rslib treats it as a static asset instead of a WebAssembly ESM module.

`wasm.mode` does not apply in this case.

## Type declaration

TypeScript does not provide built-in module declarations for `.wasm` files. If you use TypeScript, add a declaration file next to the `.wasm` file using the `.d.wasm.ts` extension, and enable [`allowArbitraryExtensions`](https://www.typescriptlang.org/tsconfig/allowArbitraryExtensions.html) in `tsconfig.json`:

```ts title="src/add.d.wasm.ts"
export function add(a: number, b: number): number;
```
number, b: number): number;
```

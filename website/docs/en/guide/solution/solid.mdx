# Solid

In this document, you will learn how to build a Solid component library using Rslib. You can check out Solid related example projects in [Examples](https://github.com/rstackjs/rstack-examples/tree/main/rslib).

## Create Solid project

You can use `create-rslib` to create a project with Rslib + Solid. Just execute the following command:

import { PackageManagerTabs } from '@theme';

<PackageManagerTabs
  command={{
    npm: 'npm create rslib@latest',
    yarn: 'yarn create rslib',
    pnpm: 'pnpm create rslib@latest',
    bun: 'bun create rslib@latest',
  }}
/>

Then select `Solid` when prompted to "Select template".

## Version status

The built-in Solid template targets Solid v1, which is the stable version used by most production Solid component libraries.

| Solid version | Rsbuild plugin | Status |
| ------------- | -------------- | ------ |
| `solid-js@^1` | `@rsbuild/plugin-solid@^1.2.2` | Stable and used by the default Rslib Solid template. |
| `solid-js@2.0.0-beta` + `@solidjs/web@2.0.0-beta` | `@rsbuild/plugin-solid@2.0.0-beta.0` | Experimental. Solid 2 and the Rsbuild Solid v2 plugin are both beta releases. |

The output shape is the same for Solid v1 and v2: compile one ESM output for general consumers and publish one preserved JSX output through the `"solid"` export condition. Solid v2 mainly differs in dependency ranges and runtime packages.

## Use Rslib in an existing project

To develop a Solid library, you need to set the [target](/config/rsbuild/output#outputtarget) to `"web"` in `rslib.config.ts`. This is crucial because Rslib sets the `target` to `"node"` by default.

To compile Solid JSX, you need to register both the [@rsbuild/plugin-babel](https://rsbuild.rs/plugins/list/plugin-babel) and [@rsbuild/plugin-solid](https://rsbuild.rs/plugins/list/plugin-solid) plugins. The Solid plugin uses `babel-preset-solid` to transform JSX into Solid's runtime DOM expressions.

```json title="package.json"
{
  "devDependencies": {
    "@rsbuild/plugin-solid": "^1.2.2",
    "solid-js": "^1.9.9"
  },
  "peerDependencies": {
    "solid-js": "^1.8.0"
  }
}
```

```ts title="rslib.config.ts"
import { pluginBabel } from '@rsbuild/plugin-babel';
import { pluginSolid } from '@rsbuild/plugin-solid';
import { defineConfig } from '@rslib/core';

export default defineConfig({
  source: {
    entry: {
      index: ['./src/**'],
    },
  },
  lib: [
    {
      bundle: false,
      dts: true,
      format: 'esm',
      plugins: [
        pluginBabel({
          include: /\.(?:jsx|tsx)$/,
        }),
        pluginSolid(),
      ],
    },
  ],
  output: {
    target: 'web',
  },
});
```

## Dual output with "solid" condition

Solid component libraries conventionally provide both a **compiled** output and a **source JSX** output. The `"solid"` export condition in `package.json` points to the uncompiled JSX source, allowing downstream bundlers (with `@rsbuild/plugin-solid`) to compile it locally with optimal settings for their target (DOM vs SSR).

Add a second lib entry that preserves JSX. The first lib entry compiles JSX via `babel-preset-solid`, while the second keeps the original JSX intact by setting SWC's `runtime: 'preserve'` and enabling Rspack's JSX parser:

```ts title="rslib.config.ts"
import { pluginBabel } from '@rsbuild/plugin-babel';
import { pluginSolid } from '@rsbuild/plugin-solid';
import { defineConfig } from '@rslib/core';

export default defineConfig({
  source: {
    entry: {
      index: ['./src/**'],
    },
  },
  lib: [
    {
      bundle: false,
      dts: true,
      format: 'esm',
      plugins: [
        pluginBabel({
          include: /\.(?:jsx|tsx)$/,
        }),
        pluginSolid(),
      ],
    },
    {
      bundle: false,
      format: 'esm',
      output: {
        distPath: {
          root: './dist/source',
        },
        filename: {
          js: '[name].jsx',
        },
      },
      tools: {
        swc: {
          detectSyntax: 'auto',
          jsc: {
            transform: {
              react: {
                runtime: 'preserve',
              },
            },
          },
        },
        rspack: {
          module: {
            parser: {
              javascript: {
                jsx: true,
              },
            },
          },
        },
      },
    },
  ],
  output: {
    target: 'web',
  },
});
```

- The first lib entry uses `pluginBabel` + `pluginSolid` to compile JSX into Solid's runtime calls (e.g. `template()`, `insert()`, `delegateEvents()`), producing standard `.js` files.
- The second lib entry skips Solid compilation entirely. `jsc.transform.react.runtime: 'preserve'` tells SWC to leave JSX syntax as-is, and `module.parser.javascript.jsx: true` tells Rspack's parser to accept JSX in the output.

Then configure `package.json` exports accordingly:

```json title="package.json"
{
  "exports": {
    ".": {
      "solid": "./dist/source/index.jsx",
      "types": "./dist/index.d.ts",
      "import": "./dist/index.js"
    }
  }
}
```

When a downstream app uses `@rsbuild/plugin-solid`, the `"solid"` condition is automatically added to module resolution, so the bundler resolves the `.jsx` source and compiles it locally. Apps that don't use a Solid-aware bundler will fall back to the pre-compiled `"import"` entry.

## TypeScript

For Solid v1 projects using TypeScript, set `"jsx": "preserve"` and `"jsxImportSource": "solid-js"` in your `tsconfig.json`:

```json title="tsconfig.json"
{
  "compilerOptions": {
    "jsx": "preserve",
    "jsxImportSource": "solid-js"
  }
}
```

For Solid v2 beta projects, the JSX runtime lives in `@solidjs/web`, so set `jsxImportSource` to `@solidjs/web`:

```json title="tsconfig.json"
{
  "compilerOptions": {
    "jsx": "preserve",
    "jsxImportSource": "@solidjs/web"
  }
}
```

## Use Solid v2 beta

Solid v2 and `@rsbuild/plugin-solid@2.0.0-beta.0` are currently beta releases. The default Rslib Solid template does not target v2, but you can adapt it manually when you intentionally want to test the beta runtime.

Change the dependency ranges in `package.json`:

```json title="package.json"
{
  "devDependencies": {
    "@rsbuild/plugin-solid": "2.0.0-beta.0",
    "@solidjs/web": ">=2.0.0-beta.0 <2.0.0-experimental.0",
    "solid-js": ">=2.0.0-beta.0 <2.0.0-experimental.0"
  },
  "peerDependencies": {
    "@solidjs/web": ">=2.0.0-beta.0 <2.0.0-experimental.0",
    "solid-js": ">=2.0.0-beta.0 <2.0.0-experimental.0"
  }
}
```

For TypeScript projects, also change `tsconfig.json`:

```json title="tsconfig.json"
{
  "compilerOptions": {
    "jsx": "preserve",
    "jsxImportSource": "@solidjs/web"
  }
}
```

If your library or examples render to the DOM directly, import DOM runtime APIs from `@solidjs/web` instead of `solid-js/web`:

```ts
import { render } from '@solidjs/web';
```

The Rslib build configuration can stay the same. `@rsbuild/plugin-solid@2.0.0-beta.0` still uses Babel and `babel-preset-solid`; the v2 difference is the compiler/runtime dependency set.

## SSR

Most Solid component libraries do not publish a separate SSR build by default. Instead, downstream frameworks resolve the `"solid"` condition, get the preserved JSX output, and compile it for their own target.

If you intentionally need a pre-compiled SSR output, pass the `solid` option to `pluginSolid()` in an additional lib entry:

```ts
pluginSolid({
  solid: {
    generate: 'ssr',
    hydratable: true,
  },
})
```
